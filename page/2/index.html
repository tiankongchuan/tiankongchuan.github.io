MeasureSpec是View的内部类，其封装了一个 View 的规格尺寸，包括 View 的宽和高的信息，它的作用是在 Measure 流程中，系统会将Vie 的LayoutParams 根据父容器所施加的规则转换成对应的 MeasureSpec，然后在onMeasure方法中根据这个MeasureSpec 来确定View的宽和高。

从 MeasureSpec 的常量可以看出，它代表了32位的int值，其中高2位代表了SpecMode，低30位则代表SpecSize。SpecMode指的是测量模式，SpecSize指的是测量大小。SpecMode 有3种模式，如下所示。

  • UNSPECIFIED：未指定模式，View想多大就多大，父容器不做限制，一般用于系统内
    部的测量。
  • AT_MOST：最大模式，对应于wrap_comtent 属性，子View 的最终大小是父View 指定的 SpecSize 值，并且子View 的大小不能大于这个值。
  • EXACTLY：精确模式，对应于 match_parent 属性和具体的数值，父容器测量出 View
    所需要的大小，也就是SpecSize的值。

view的MeasureSpec

实际上，view的MeasureSpec是由父容器和自己LayoutParams共同决定的。根本就不是自己测量出来的。这打破了出入门时一个直观的认知。

当我们调用Activity的startActivity方法时，最终是调用ActivityThread的 handleLaunchActivity方法来创建Activity的。
在该方法中，调用performLaunchActivity方法来创建Activity，在这里面会调用到
Activity的onCreate方法，从而完成DecorView的创建（此处不要被误导了，还是在setContentView中创建的）
进一步的，调用handleResumeActivity，在此方法中：

  1. 调用Activity的onResume方法
  2. 得到DecorView
  3. 得到WindowManager,WindowManager是一个接口并且继承了接口ViewManager。
  4. 调用WindowManager的addView方法，WindowManager的实现类是WindowManagerImpl，所以实际调用的是WindowManagerImpl的addView方法。在WindowManagerImpl的addView 方法中，又调用了 WindowManagerGlobal的addView方法，在该方法中调用了ViewRootImpl的setView方法并将DecorView作为参数传进去，这样就把DecorView加载到了Window。

Drawable的内部宽高是两个重要参数，用以下两个函数可以获得：

  • getIntrinsicWidth
  • getIntrinsicHeight
    并非所有Drawable都有内部宽高，如果是一张图片形成的Drawable，内部宽高就是图片的大小；如果是颜色形成的Drawable，则没有此概念。Drawable的内部宽高不等于其大小，一般而言，Drawable没有大小概念，作为View的背景时，Drawable会被拉伸到View的同等大小。

BitmapDrawable

展示一张图片

ShapeDrawable

通过颜色来构造图形，可以是纯色的图像，也可以是具有渐变效果的图形。

LayerDrawable

对应的标签是<layer-list>，它表示一种层次化的Drawable集合。可以实现将不同的Drawable放置在不同的层上从而达到一种层叠的效果。

StateListDrawable

对应于<selector>，它也表示Drawable集合，每个Drawable都对应View的一种状态。系统可以根据View的状态来选择合适的Drawable。
系统会根据View的状态从selector中选择对应的item，从上到下直到找到第一条匹配的item。一般而言，默认的item都应该放在selector最后一条并且不附带任何的状态，这样如果往上所有item都无法匹配View的当前状态，就会选择默认的item，因为它不附带任何状态，所以它可以匹配任何状态。

LevelListDrawable

对应<level-list>标签，同样表示一个Drawable集合，集合中每个Drawable都有一个对应的等级。根据不同的等级LevelListDrawable会切换为对应的Drawable。

TransitionDrawable

对应<transition>，用于实现两个Drawable之间的淡入和淡出。

InsetDrawable

对应<insert>标签，它可以将其他Drawable内嵌到自己当中，并且可以在四周流出一定的间距。

ScaleDrawable

对应<scale>标签，它可以根据自己的等级将指定的Drawable缩放到一定比例。

ClipDrawable

可以根据自己当前的等级来裁剪另一个Drawable。

创建型设计模式

单例模式

大部分场景下，使用静态内部类单例模式就可以了

它的优点如下：

  1. 懒加载（Lazy Initialization）： 静态内部类只有在被引用的时候才会被加载，因此在首次调用 getInstance() 方法时才会创建实例。这实现了懒加载的效果，即只有在需要时才会创建单例对象，节省了资源。
  2. 线程安全： 静态内部类在加载时进行了同步，因此在多线程环境下，保证了线程安全。这是因为类加载过程是线程安全的，并且只有在类加载完成后，才会创建静态内部类的实例。
  3. 不需要额外的同步锁： 由于静态内部类的加载是由 JVM 进行管理的，因此不需要额外的同步锁来保证线程安全，从而避免了性能问题。
  4. 延迟初始化： 单例对象在静态内部类中被初始化，这实现了延迟初始化的效果。在单例模式中，有时候并不需要在应用启动时就创建实例，而是在需要的时候再创建，从而提高了应用的启动性能。
  5. 简洁性： 这种写法相对简洁，封装了实例的创建逻辑，隐藏了具体的实现细节。

动态加载技术

传统的安卓开发中，开发完之后的app上架到各个渠道之后，就不能修改源码了。只能通过服务器来控制应用中预留的分支代码。但是很多时候我们无法提前预知需求和突然发生的情况，也就不能提前在应用代码中预留分支代码，这时就需要采用动态加载技术。在应用程序运行时，动态加载一些程序中原本不存在的可执行文件并运行这些文件里的代码逻辑。可执行文件总的来说分为两种：

  1. 动态链接库so ，
  2. dex相关文件（dex以及包含dex的jar/apk文件）

动态加载技术进一步的演化成了：

  1. 热修复技术
  2. 插件化技术

本地

  1. 仓库的概念：自己在文件夹中git init之后的，这个文件夹就可以被视为一个仓库了。去github上看别人的文件其实也就是进了别人的仓库。
  2. 自己新建的仓库，所有文件都还是没有被跟踪的。在生成一个版本之后，没有被跟踪的文件就不会在这个版本里，自然也就无法记录什么时候被修改之类的信息了。所以要先跟踪一个文件或者一整个目录，使用git add <name> 这个命令即可。对于目录而言，*.* 代表当前目录，如果不想跟踪了，使用git rm <name> 不让他继续跟踪了。或者使用git rm -cache<name> 不让它跟踪了，但是保留在目录中。

    或者将其设置为仓库之后，直接进行一次git commit 命令上传一次，其实也就有了第一个版本，文件里面的内容也都自动变成了跟踪状态了。

  3. 比如已经完成了对一个程序的修改，就可以使用git add <name> 把某个文件的状态设置为缓存状态了，如果是手滑不想将其设置为缓存状态，使用git reset HEAD <name> 来取消它的缓存状态。
  4. git commit 提交这一次的修改。一般来说每次提交是要一些附带的提交信息的，我们可以使用git commit -m ‘附加信息’ 来进行提交。如果不想提交了，使用git reset head~ –soft来取这次提交。git commit -a可以直接把没有暂存的文件也提交了
  5. 使用git status来查看文件的状态，但是这个命令只能看个大概，比如是不是暂存了，或者已经提交了之类的。如果想看更详细的，使用git diff，进一步的，使用git log来看历史的提交。

远程

  1. 找到一个远程仓库（自己取github建就可以了），然后使用git remote add 远程仓库名（自己取） 仓库地址。再用git remote就可以查看一下了。
  2. 要把本地的代码推到仓库里，使用git push 远程仓库名 master

分支

每次提交生成一个新版本的时候，都会生成一个提交对象，每一个提交对象，都有一个独一无二的哈希值。所谓的分支，就是包含哈希值的一个文件，可以把他理解为指向一个提交对象的指针。

使用git branch –list来看自己当前在哪个分支上，前面带*的就是当前分支。
使用git branch 分支名就可以创建一个新的分支
使用git checkout 分支名可以到指定分支上（好像当前要用switch切换了）
使用git checkout -b 分支名可以创建并且直接切换过去
使用git merge 分支名将这个分支合并到master上去

其他

git stash储藏当前内容，应对需要切换出去的时候保存当前内容
git stash apply恢复当前内容

冷启动和热启动

冷启动（Cold Start）：
冷启动指的是当你首次启动一个应用程序时发生的情况，或者在应用程序已经被系统完全销毁（如由于内存不足而被系统回收）后再次启动应用程序的情况。在冷启动时，应用程序的进程会重新创建，并且需要进行一系列初始化操作，如加载类、创建对象、初始化资源等。这通常会导致启动耗时较长，因为涉及到较多的初始化工作。
冷启动的性能问题可能会影响用户体验，因为用户可能会感受到应用程序的启动速度较慢。为了改善冷启动性能，开发者可以采取一些策略，如延迟加载、异步初始化、使用Splash Screen 等，以减少用户等待时间。

热启动（Warm Start）：
热启动是指在应用程序已经在后台运行，并且进程仍然存在的情况下，再次启动应用程序的情况。在热启动时，应用程序的进程仍然保持，部分初始化操作已经完成，因此启动速度会比冷启动快很多。应用程序可能从后台状态恢复，或者重新显示之前的界面。
热启动的性能通常比冷启动好，因为它不需要重新创建进程和进行全面的初始化操作。然而，一些状态可能需要重新加载，例如数据的刷新或恢复。
总结起来，冷启动是指首次启动应用程序或完全重新启动应用程序时的情况，而热启动是指在应用程序已经在后台运行的情况下再次启动应用程序。了解这两个概念可以帮助开发者更好地优化应用程序的启动性能和用户体验。

Gradle 是一个构建工具，用于编译、构建和打包 Android 应用程序。Android 插件是与 Gradle 配合使用的插件，专门用于构建和管理 Android 项目。这些组件之间存在一些关系和约束。

  1. Gradle 版本与 Android 插件版本关系： 不同版本的 Android 插件对 Gradle 的版本有特定的要求。通常，在使用新版本的 Android 插件之前，需要确保你的 Gradle 版本满足最低要求。你可以在官方文档中找到 Android 插件与 Gradle 版本之间的兼容性表格，确保你选择的 Android 插件版本支持你所使用的 Gradle 版本。

  2. Android 插件版本与 Android 版本关系： Android 插件版本决定了你可以在 Android 项目中使用的功能和构建选项。较新的 Android 插件通常支持更多的功能和新的 Android 版本特性。因此，如果你希望在项目中使用最新的 Android 功能，你需要使用相应的 Android 插件版本，并确保你的 targetSdkVersion 和 compileSdkVersion 对应到该版本所支持的 Android 版本。

  3. Android 版本与 SDK 版本关系： Android 版本是指 Android 操作系统的版本，而 SDK (Software Development Kit) 版本则是指用于开发 Android 应用程序的开发工具集版本。每个 Android 版本都对应一个特定的 SDK 版本，这个 SDK 包含了开发应用所需的 API 和工具。你需要选择一个适合你的应用程序目标受众和功能需求的 Android 版本，并使用相应的 SDK 版本进行开发。

  4. SDK 版本与 JDK 版本关系： JDK (Java Development Kit) 是用于开发 Java 应用程序的工具集。Android 应用程序的开发依赖于 Java 编程语言，因此你需要安装 JDK。与 SDK 版本相比，JDK 版本在 Android 开发中的选择没有那么严格的要求，因为 Android 应用程序并不直接依赖于 JDK 版本。然而，为了获得最佳性能和兼容性，建议使用较新的 JDK 版本。

约束关系总结：

  • Gradle 版本要与所使用的 Android 插件版本兼容。
  • Android 插件版本要与你的目标 Android 版本兼容，并决定了你可以使用的构建选项和功能。
  • 你的 targetSdkVersion 和 compileSdkVersion 应与所使用的 Android 插件版本相匹配。
  • SDK 版本是你用于开发 Android 应用程序的工具集，需要与你的目标 Android 版本兼容。
  • JDK 版本是你用于开发 Android 应用程序的 Java 开发工具，建议使用较新的版本以获得更好的性能和兼容性。

  1. Gradle版本和AGP版本的约束：

    • 不同版本的AGP可能需要特定版本的Gradle进行构建。通常，在升级AGP版本时，你需要检查官方文档或发布说明，以找到与该AGP版本兼容的Gradle版本。在项目的根目录的build.gradle文件中，你会看到类似于com.android.tools.build:gradle:4.2.0的声明，其中的4.2.0就是对应的AGP版本。
    • AGP版本和Gradle版本的对应关系可以在Android开发者网站或Android开发者工具的官方文档中找到。确保使用匹配的Gradle版本和AGP版本是确保项目可以成功构建的关键。

  2. AGP版本和Android SDK版本的约束：

    • AGP版本通常支持一定范围的Android SDK版本。比如，AGP版本4.2.0可能支持Android SDK版本21到31。你可以在AGP版本的发布说明或官方文档中找到支持的Android SDK版本范围。
    • 在项目的build.gradle文件中，你可以通过compileSdkVersion和targetSdkVersion来指定项目使用的Android SDK版本。compileSdkVersion指定编译时使用的SDK版本，而targetSdkVersion指定应用程序目标运行的Android版本。确保compileSdkVersion和targetSdkVersion与你选择的AGP版本兼容。

  3. Android SDK版本和目标Android版本的约束：

    • compileSdkVersion指定的Android SDK版本应该足够包含你在项目中使用的所有API和功能。这意味着你需要选择一个兼容目标设备的SDK版本。
    • targetSdkVersion指定应用程序目标运行的Android版本。确保应用程序在该目标版本上的功能和行为符合预期。
