实际上就是讨论Message、MessageQuene、Looper和Handle组成的消息循环。因为安卓的消息机制实际上就是Hndler的运行机制。

本质上来说，Handle的作用是将一个任务切换到某个指定的线程中去执行。

消息队列

也就是MessageQueue，虽然叫做队列，但是实际上它的实现是一个单链表。
其中它读取消息的方法叫做next方法，它是一个无限循环的方法，如果队列中没有消息，那么next就会一直堵塞在这里，有新消息来到之后，next方法就会返回这条消息并将其从单链表中移除。

Looper工作原理

准备工作

如果我们希望使用Handler，那么当前线程中就必须要有一个Looper。可以通过Looper.prepare() 来为当前线程创建一个Looper，然后通过Looper.loop() 来开启消息循环。
其中主线程比较特殊的是，提供了prepareMainLooper() 方法来创建主线程中的Looper，其本质还是使用prepare。同时提供了getMainLooper 的方法，可以在任何地方获取到主线程的Looper。

之所以要大费周章的创建和获取，是因为Looper是可以停止的，通过quit和quitSafely 即可让整个Looper停止，前者会直接退出，后者会将已有消息处理完之后再安全地退出。

工作流程

消息队列中提到了next，是一个无限循环的方法，同时也会有返回值，如果有消息了就返回这个消息，没有消息就阻塞。实际上，是Looper一直在调用这个方法，如果next阻塞了那么Looper也会阻塞在这里。
所以要停止Looper的方法也很简单了，让next返回null值即可。

多进程可能导致的问题

  1. 静态成员和单例模式完全失效
  2. 线程同步机制完全失效
  3. SharedPreferences的可靠性下降
  4. Application会多次创建

对于安卓系统而言，对于不同的进程，都会为其分配一个独立的虚拟机，而不同的虚拟机在内存分配上会有不同的地址空间。所以用的内存都不是同一块了，一些共享的问题就会随之出现。

IPC基础概念

Serializable接口

java提供的序列化接口，是一个空接口，为对象提供标准的序列化和反序列化操作。直接继承它，然后在类中指定一个标识值即可。
![[Pasted image 20230807112324.png]]

简单来说，系统在序列化的时候会把这个标识符写到序列化的文件中去，反序列化的时候会拿出来与类的标识符进行对比，如果不对的话，则无法正常反序列化。

如果不指定该值，并且在反序列化的时候这个类增加或者删除了某些变量，那么系统会重新计算该类的值。如此一来，反序列化就会失败。
相反如果指定了话(手动指定，或者让IDE通过类的结构来计算这个值)，上述问题就不会出现了。

Parcelable

实现了该接口的数据就可以在Binder中自由的传输了，需要实现三个功能：

  1. 序列化：由writeToParcel完成，最终是通过一系列的write方法来完成的
  2. 反序列化：由CREATOR方法完成，内部要表明如何创建序列化对象和数组，最终是通过一系列的read方法实现的
  3. 内容描述：由describeContents方法完成，几乎所有情况下该方法都应该返回0，仅当当前对象中存在文件描述符的时候返回1。

![[Pasted image 20230807141414.png]]

安卓中实现了该接口的类包括：Intent，Bundle，Bitmap等。

IPC具体方式

使用Bundle

它实现了Parcelabel接口，所以可以方便地在不同的进程中传输。
当然，传输的数据也必须要是能够被序列化的，比如基本数据类型，实现了两个接口中任意一个的对象，和一些安卓支持的特殊对象。

文件共享

非常直观的方法，但是需要考虑并发读写的问题。

使用Messenger

可以通过它在不同的进程中传递Message对象，其底层实现是AIDL。

AIDL

AIDL是上述Messenger的底层。在Messenger中，它是以串行的方式处理客户端发送过来的消息的，如果大量消息发送到服务端，那么再使用Messenger就不合适了。

AIDL支持的类型

  1. 基本数据类型
  2. String 和 CharSequence
  3. ArrayLsit，其中每个元素都要被AIDL支持
  4. HashMap，其中每个元素都要被AIDL支持，包括key和value
  5. Parcelable
  6. AIDL

不管是Activity、Dialog还是Toast，它们的视图都是附加在Window上面的，所以Window是View的直接管理者。
外界访问Window，都是通过WindowManager，这相当于是访问的入口。
Winodw的具体实现，在WindowManagerServer中。
WindowManager和WindowManagerServer的交互是一个IPC过程。

Window的层级

  1. 应用Window：对应一个Activity
  2. 子Window：不能独立存在，需要附属在特定的父Window中，比如Dialog
  3. 系统Window：需要声明权限才能创建的Window，比如Toast和系统状态栏

Window是分层的，层级大的会覆盖在层级小的Window上面。在上述三类Window中，应用Window层级范围是1-99，子Window的层级范围是1000-1999，系统Winodw的层级范围是2000-2999。

提供的方法

一般只有三个方法：添加、更新、删除View。
这三个方法是在ViewManager接口中实现的，而WindowManager实现了这个接口。

Window的内部机制

开门见山的说，Window是一个抽象的概念。
每一个Window都有一个View和ViewRootImpl，Window和View通过ViewRootImpl来建立联系。

Measure

总的来说，measure用来测量View的宽和高，它的流程分为View的measure流程和 ViewGroup的measure 流程，只不过ViewGroup的measure流程除了要完成自己的测量，还要遍历地调用子元素的measure()方法。

View的measure

对于一个直接继承自View的自定义View来说，它的wrap_content和match_parent属性的效果是一样的。因此如果要实现自定义View的wrap_content，则要重写onMeasure方法，并对自定义View的wrap_content属性进行处理。
而在UNSPECIFIED模式下返回的是getDefaultSize方法的第一个参数size的值，size的值从onMeasure方法来看是getSuggestedMinimumWidth方法或者getSuggestedMinimumHeight方法得到的。简单来说，如果没有设置背景，那就默认0（可以更改），如果有背景，就在默认值和背景的大小之间取最大值。

ViewGroup

对于ViewGroup，它不只要测量自身，还要遍历地调用子元素的measure()方法。ViewGroup中没有定义onMeasure()方法，但却定义了measureChildren()方法。在此方法中，会根据父容器的MeasureSpec模式再结合子元素的LayoutParams 性来得出的
子元素的MeasureSpec属性。。
ViewGroup并没有提供onMeasure方法，而是让其子类来各自实现测量的方法，究
其原因就是ViewGroup有不同布局的需要，很难统一。

Layout

ViewGroup 中的 layout 方法用来确定子元素的位置，View中的layout方法则用来确定自身的位置。

View

setFrame方法用传进来的l、t、r、b这4个参数分别初始化mLeft、mTop、mRight、mBottom这4个值，这样就确定了该View在父容器中的位置。
在setChildFrame 方法中调用子元素的layout方法来确定自己的位置。

view

  2. 保存当前 canvas 层。
  3. 绘制 View 的内容。
  4. 绘制子 View。
  5. 如果需要，则绘制 View 的褪色边缘，这类似于阴影效果。
  6. 绘制装饰，比如滚动条。

继续而言，此时DecorView中的内容还是没有展示的，因为它还没有被加载到window中。
此处需要了解[[Activity的创建]]，在此过程中了解Decor是什么怎样加载到window中的。
提到的ViewRootlmpl，还有一个PerformTraveals方法，它开始View的工作流程，里面会调用三个方法：

  • performMeasure
  • performLayout
  • performDraw
    进而分别调用view的measure，layout和draw方法，开始View的工作流程。而值得注意的是，performMeasure有两个参数，分别是childWidthMeasureSpec和childHeightMeasureSpec。要了解这两个参数，需要[[MeasureSpec了解]]。

进一步的，就是对[[View的三个流程]]的解析。

View的事件分发

点击屏幕，就会产生一个事件，这个事件被封装成了类，即MotionEvent。该类在View中的层级传递过程就是点击事件分发。进一步的，有三个重要方法：

  • dispatchTouchEvent(MotionEvent ev)—用来进行事件的分发。
  • onInterceptTouchEvent(MotionEvent ev)—用来进行事件的拦截，在dispatchTouchEvent()中调用，需要注意的是 View 没有提供该方法。
  • onTouchEvent(MotionEvent ev)—用来处理点击事件，在 dispatchTouchEvent()方法中进
    行调用。

传递

点击屏幕后，首先是Activity调用dispatchTouchEvent方法，而Activity会让PhoneWindow处理，而PhoneWindow再交给DecoView，DecoView再交给其根ViewGroup来进行进一步的操作。

分发

一般而言，ViewGroup是不会拦截事件的。会进入一个循环 ，倒序遍历ViewGroup中的子元素，如果有子元素能够接收到点击事件，则将事件交给它处理。(通过判断触摸点是否在View的范围内或者子View是否在播放动画)

什么时候才能获取view大小

明确地说，在onCrete、onStart、onMeasure方法中，都是不能正确获取到view的宽高信息的。因为measure的过程与Activity的生命周期并不同步。如果想要获取，有以下四种方式：

  1. onWindowFocusChanged
    需要注意的是，这个方法会在Activity的窗口得到和失去焦点的时候被调用，这意味着该方法是会被反复调用的，换言之，如果你的onResume和onPause调用很频繁，那么这个函数的调用也会很频繁，在这个函数内容处理逻辑的时候要考虑这种情况的发生。
  2. view.post(runnable)
    将消息投递到消息队列的尾部，此时的View已经初始化好了。
  3. ViewTreeObserver
    其中有很多的回调可以完成，比如使用onGlobalLayoutListener，当view树状态改变或者view树的view可见性发生改变的时候，onGlobalLayout方法会被回调，这是一个获取宽高的好时机，但是它也会有被调用多次的可能性，考虑这种情况对逻辑的影响。
